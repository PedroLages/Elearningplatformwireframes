# Code Review: E04-S01 — Mark Content Completion Status

**Date:** 2026-03-02
**Reviewer:** code-review agent (Opus)

## What Works Well

1. **Clean store architecture** — `useContentProgressStore` follows the established `persistWithRetry` + optimistic-update pattern from `useBookmarkStore`. Flat `statusMap` with O(1) lookups.
2. **Well-structured parent cascade** — `deriveModuleStatus` correctly handles all three cases and empty-lessons edge case. Synchronous cascade within `setItemStatus` prevents stale parent UI.
3. **Good component composition** — StatusIndicator and StatusSelector properly separated, composed via Radix Popover. `data-status` and `data-testid` enable reliable E2E.

## Findings

### Blockers

**B1 (confidence: 92):** `useContentProgressStore.ts:99-111` — Multiple `db.contentProgress.put()` calls not wrapped in a Dexie transaction. Item + cascade records can be partially persisted if a put fails mid-sequence. AC2 requires "no partial or inconsistent state."

**B2 (confidence: 90):** `ModuleAccordion.tsx:93-99` — Module-level `StatusIndicator` is a `<button>` nested inside `AccordionTrigger` (also a `<button>`). Nested interactive elements violate HTML spec and WCAG. `stopPropagation()` prevents toggle but doesn't fix DOM nesting.

### High Priority

**H1 (confidence: 88):** `schema.ts:21` — `EntityTable<ContentProgress, 'courseId'>` is incorrect for compound key `[courseId+itemId]`. TypeScript type is misleading — `get(courseId)` compiles but fails at runtime.

**H2 (confidence: 85):** `ModuleAccordion.tsx:46` — `useContentProgressStore()` called without selector subscribes entire component to all store changes. Every status change re-renders all modules/lessons.

**H3 (confidence: 92):** No unit tests for `useContentProgressStore`, `StatusIndicator`, or `StatusSelector`. Critical business logic (optimistic update, rollback, cascade) only tested via E2E.

**H4 (confidence: 80):** `story-e04-s01.spec.ts:78-91` — AC2 E2E test verifies optimistic update but NOT IndexedDB persistence. No reload-and-verify step.

### Medium

**M1 (confidence: 75):** `StatusIndicator.tsx:42-60` — Touch target ~20x20px from `p-0.5` padding. Below 44x44px WCAG minimum.

**M2 (confidence: 72):** `StatusIndicator.tsx:21-32` — Uses `text-blue-500`/`text-green-500` instead of theme tokens (`blue-600`/`green-600`).

**M3 (confidence: 78):** `ModuleAccordion.tsx:148-155` — Manual `onClick` + `PopoverTrigger asChild` creates double click handling. Fragile pattern.

**M4 (confidence: 70):** `useContentProgressStore.ts:51-64` — `loadCourseProgress` merges via spread, causing unbounded accumulation across courses.

### Nits

**N1:** `StatusSelector.tsx:10` — Icon field abstraction adds indirection for 3 hardcoded options.
**N2:** `StatusIndicator.tsx:46` — Unnecessary template literal in aria-label.
**N3:** `ModuleAccordion.tsx:42` — Dead `_completedLessons` prop kept for backward compat.
